## S5 — End-to-end on the v1.0 personas (excluding FIDO2)

| Field | Value |
|---|---|
| **PRD §11 criterion** | A developer in any of the four v1.0 problem scenarios (multi-account, bastion, GHE-with-CA, hardware key) can clone, sign a commit, and push using only `git` + `gitway`.  Note: the FIDO2 / hardware-key persona depends on M16 which is post-1.0 deferred; S5 for v1.0 is scoped to the three remaining personas. |
| **Tooling** | Manual smoke test, recorded below. |
| **Status** | ⏳ Pending (M20.0b — manual gate before M20.6 Gitway 1.0 tag). |

### Persona 1 — Multi-account

```sh
# Two accounts, each with its own key, routed via Host blocks.
cat ~/.ssh/config
# Host github-personal
#     Hostname github.com
#     User git
#     IdentityFile ~/.ssh/id_ed25519_personal
# Host github-work
#     Hostname github.com
#     User git
#     IdentityFile ~/.ssh/id_ed25519_work

git clone github-personal:user/personal-repo.git
cd personal-repo && git -c gpg.format=ssh -c gpg.ssh.program=gitway-keygen \
  -c user.signingkey=~/.ssh/id_ed25519_personal.pub \
  commit --allow-empty -m "test" -S
git push
```

- [ ] Personal account clone OK
- [ ] Personal account signed commit OK
- [ ] Personal account push OK
- [ ] Work account clone OK
- [ ] Work account signed commit OK
- [ ] Work account push OK

### Persona 2 — Bastion (jump host)

```sh
gitway --jump-host bastion.example -- git clone git@internal.example:repo.git
gitway config show internal.example  # verify chain visualization
```

- [ ] Two-hop fetch OK
- [ ] `gitway config show` renders the chain

### Persona 3 — GHE-with-CA

```sh
# Add a @cert-authority line in ~/.ssh/known_hosts pointing at the
# org CA's public key.  Confirm gitway accepts the GHE host key
# signed by that CA (FR-60 / FR-64 — the partial M14 path).
gitway --test ghe.example.com
```

- [ ] CA-signed host key accepted
- [ ] FR-60 acceptance via known_hosts confirmed
- [ ] FR-64 `@revoked` check runs first (verify by testing a revoked
      fingerprint blocks even with `StrictHostKeyChecking=no`)

## S7 — `-vvv` parity with OpenSSH

| Field | Value |
|---|---|
| **PRD §11 criterion (new)** | A connection failure to a deliberately misconfigured GHE test instance produces a `gitway -vvv` output that surfaces the same root-cause information as `ssh -vvv` against the same target. |
| **Tooling** | Manual side-by-side comparison. |
| **Status** | ⏳ Pending (M20.0b — manual gate before M20.6 Gitway 1.0 tag). |

```sh
ssh -vvv -p 2222 git@ghe.example.com 2>&1 | tee ssh-vvv.log
gitway -vvv --port 2222 --test ghe.example.com 2>&1 | tee gitway-vvv.log
diff <(grep -E 'kex|host key|auth' ssh-vvv.log) \
     <(grep -E 'CAT_KEX|CAT_AUTH|CAT_CONFIG|CAT_RETRY' gitway-vvv.log)
```

- [ ] Both runs surface the same root-cause field (KEX algorithm
      mismatch / host-key mismatch / auth method mismatch).

## Automated smoke against the released v1.0.0 binary (2026-05-05)

After the v1.0.0 tag was pushed and the release CI completed,
the local Windows MSVC build (`target/release/gitway.exe` —
9.15 MB, links `anvil-ssh = "1.0.0"` from crates.io) was
exercised against `github.com` to validate the shipped binary
end-to-end.

This is **not** a substitute for the persona gates above — it
is a binary-level sanity check.  The persona gates still need a
human run.

| Smoke check | Result |
|---|---|
| `gitway --version` | ✅ `gitway 1.0.0` |
| `gitway describe` | ✅ `tool=gitway`, `schema_version=1.0.0`, `version=1.0.0`, 11 commands listed, all 6 agent env vars enumerated, 3 providers |
| `gitway schema` | ✅ `schema_version=1.0.0`, 11 commands, 2 binaries, 21 global flags, 6 exit codes (0/1/2/3/4/78) |
| `gitway --test --json github.com` | ✅ Connected on port 22; `host_key_verified=true`; fingerprint `SHA256:+DiY3wvvV6TuJJhbpZisF/zLDA0zPMSvHdkr4UvCOqU` matched the embedded pin; `authenticated=true`; envelope shape conforms to `docs/json-schema.md` (metadata + data, schema_version present) |
| `gitway -vvv --test github.com` (S7 surface check) | ✅ 13 structured tracing events at `anvil_ssh::auth`, `anvil_ssh::session`, `anvil_ssh::kex` targets — confirms the diagnostic depth surface is wired in the shipped binary |
| `gitway --port 1 --connect-timeout 1 --attempts 3 ... --test github.com` (M18 retry machinery) | ✅ 3 attempts × ~1s timeout = 4.02 s elapsed end-to-end; error correctly classified as I/O `TimedOut`; JSON envelope on stderr has `metadata.schema_version=1.0.0`, `error.code=GENERAL_ERROR`, `error.exit_code=1` |

These confirm the **binary works, the contract surfaces are
frozen, and the retry/timeout machinery shipped correctly**.
They do not cover multi-account routing, bastion chains, GHE
CA validation, or `-vvv` parity against OpenSSH on a
deliberately-misconfigured target — those require infrastructure
beyond the dev box and stay marked ⏳ Pending below until a
human runs them.
